## 第一部分：安全验证加强

### 问题背景

原始登录验证存在以下潜在问题：
- JWT token 仅在颁发时校验，不再检查用户是否真实存在
- 被禁用的用户若 token 未过期，仍可访问管理端
- 权限信息可能过期，无法实时同步

### 改进措施

#### 1. **增强 JwtAuthGuard**（server/src/modules/auth/jwt-auth.guard.ts）

```typescript
✨ 新增功能：
- 每次请求时重新查询数据库验证用户状态
- 检查用户是否存在（防止已删除用户的 token 继续有效）
- 检查用户账号是否处于活跃状态（status === 'active'）
- 动态更新权限信息（防止权限更变后的延迟）
```

**验证流程**：
```
1. 提取 Authorization: Bearer <token>
2. 验证 token 签名与有效期
3. 查询数据库确认用户存在
4. 检查用户状态是否为 'active'
5. 加载最新角色与权限信息
6. 拒绝访问（401）：未登录、token 失效、用户不存在、账号禁用
```

**安全改进对比**：

| 检查项 | 原版 | 改进版 |
|-------|------|-------|
| Token 签名验证 | ✅ | ✅ |
| Token 有效期检查 | ✅ | ✅ |
| 用户是否存在 | ❌ | ✅ |
| 用户账号状态 | ❌ | ✅ |
| 权限信息实时性 | ❌（颁发时固定） | ✅（每次请求更新） |
| 防止已删除用户访问 | ❌ | ✅ |

#### 2. **权限守卫框架**（server/src/modules/auth/admin.guard.ts）

创建 `AdminGuard` 和 `@RequirePermission` 装饰器，为后续权限校验奠定基础：

```typescript
@UseGuards(JwtAuthGuard, AdminGuard)
@RequirePermission('content:review')
async reviewContent() { ... }
```

**后续使用**：此框架已预留，可按需应用于需要特定权限的端点。

### 测试验证

```bash
✅ 正确密码登录成功 → 返回 token
✅ 错误密码登录失败 → 401 "账号或密码错误"
✅ 有效 token 访问受保护端点 → 成功
✅ 无 token 访问受保护端点 → 401 "未登录"
✅ 过期 token 访问 → 401 "登录已失效"
✅ 已禁用用户登录后访问 → 401 "用户账号已禁用"
```

## 已知限制

### 临时已知问题（可后续改进）

1. **权限强制执行**（设计限制）
   - 当前后端仅验 JWT，未强制校验权限码
   - AdminGuard 已预留框架，可按需应用

2. **新增用户交互**（数据脚本限制）
   - enhance-seed.js 中生成的用户没有与原始内容关联
   - 建议：手动在 UI 上执行发布 + 收藏操作以验证完整流程

3. **消息通知（架构限制）**
   - 当前 `messages` 表无 userId 列，消息是全局广播
   - 改进方案：详见 启动与联调说明.md 第五节
