# 启动与联调说明

> 2026-06-11 更新：原版 `server/`（NestJS 10 + Prisma 5 + **MySQL**）已从 Mac 并入仓库并在 Windows 完成三端联调。
> 浏览器实测：H5 7 个页面 + 管理端 17 个页面控制台零报错、零失败请求；
> 「发布 → 待审 → 管理端审核 → 用户端可见 → 消息通知」闭环全通。
> （此前用于过渡的 SQLite 重建版后端备份在 `server-rebuilt-sqlite.bak/`，确认不需要后可删除。）

## 一、技术栈与端口

| 端 | 目录 | 技术栈 | 端口 |
|----|------|--------|------|
| 后端服务 | `server/` | NestJS 10 + Prisma 5 + MySQL 8 | `3000`（接口前缀 `/api`，静态文件 `/uploads`） |
| 后台管理 | `admin/` | Vue3 + Element Plus + Vite | `5181`（代理 `/api`、`/uploads` → 3000） |
| 用户端 | `frontend/` | uni-app（Vue3）H5 / 微信小程序 | `5173`（代理 `/api`、`/uploads` → 3000） |

包管理器统一使用 **pnpm**（勿在仓库根目录 `npm install`：npm workspaces 会把依赖提升到根
`node_modules`，导致 uni-app 解析到带 slot 渲染回归的新 alpha。uni-app 已锁定
`3.0.0-alpha-5000920260512001`）。

## 二、MySQL（本机开发环境）

Windows 本机使用**便携版 MySQL 8.0.44**（无需安装服务）：

- 位置：`C:\Users\Admin\AppData\Local\qingning-mysql`（删除该目录即卸载）
- 启动：`& "$env:LOCALAPPDATA\qingning-mysql\mysql-8.0.44-winx64\bin\mysqld.exe" --defaults-file="$env:LOCALAPPDATA\qingning-mysql\my.ini" --console`
- 配置要点：`collation-server = utf8mb4_unicode_ci`（**必须**——项目迁移依赖该排序规则；
  MySQL 8 默认的 `utf8mb4_0900_ai_ci` 会让 `prisma migrate dev` 的影子库重放报 1267 collation 冲突，
  Mac 上如遇同样报错，参照本配置改 `my.cnf` 或用 `prisma db push` 绕过影子库）
- 账号：`qingning / change_me`（与 `server/.env` 一致）、`user / password`、`root`（无密码，仅本机）
- 数据库：`qingning_local_life`（utf8mb4_unicode_ci）

`server/.env`（被 .gitignore 忽略，新机器需手建）：

```ini
NODE_ENV=development
PORT=3000
DATABASE_URL="mysql://qingning:change_me@127.0.0.1:3306/qingning_local_life"
JWT_SECRET="qingning-local-life-dev-secret-0611"

ADMIN_USERNAME="admin"
```

## 三、首次启动

```bash
# 1. 安装依赖（各子目录独立 pnpm）
pnpm run install:all

# 2. 启动 MySQL（见上节），然后建库迁移 + 种子（种子可重复执行，全 upsert）
cd server && npx prisma migrate dev && npm run seed

# 3. 三个终端分别启动
pnpm run dev:server     # http://127.0.0.1:3000/api
pnpm run dev:admin      # http://127.0.0.1:5181
pnpm run dev:frontend   # http://127.0.0.1:5173
```

后台账号：`admin / change_me_please`（超级管理员）。
用户端无登录体系：后端以「第一个无 username 的用户」（种子昵称：青柠用户）承载发布/收藏/投递/消息，
接入微信登录时替换 `server/src/modules/public/public.service.ts` 的 `currentUser()`。

## 五、已知设计限制（未改动，按需排期）

- **站内信无收件人**：`messages` 表没有 userId，消息是全局广播；接入登录体系时需加列迁移。
- **RBAC 未强制执行**：前端路由 meta.permissions 未在守卫校验、后端仅验 JWT 不验权限码；
  当前单管理员场景无影响，多角色上线前需补 PermissionsGuard。
- **`POST /api/uploads` 无鉴权**：用户端无登录态故必须开放，已限 image/* 且 5MB。
- **用户端列表一次性拉 100 条**（无翻页加载），数据量大后需做分页/加载更多。
- **广告 linkType=h5/miniapp 的跳转未实现**（用户端 openBanner 只支持 category）。
- AdPosition 的 pv/uv/ctr 为静态种子值，无真实埋点。

## 六、生产部署

`server/Dockerfile` 已按 MySQL 设计；部署时通过环境变量覆盖 `DATABASE_URL`/`JWT_SECRET`，
执行 `prisma migrate deploy` 应用迁移（注意目标库 collation 需为 `utf8mb4_unicode_ci`）。
