## 二、核心功能模块详解

### 模块一：内容管理（五类核心业务）

#### 1. 招聘管理
招聘是平台的流量入口，需要精细化审核和管理。

**管理端页面**：

| 页面 | 功能说明 |
|------|----------|
| 招聘列表 | 展示全部招聘信息，支持按类型筛选（全职/兼职/临时工/招工）、按状态筛选（待审/已发布/已下架/已过期）、按时间排序 |
| 招聘详情 | 查看完整职位信息，包括公司名称、岗位职责、薪资待遇、工作地点、联系人、联系电话等 |
| 招聘审核 | 每条招聘信息需经审核：通过/拒绝（拒绝需填写拒绝理由，反馈给发布者） |
| 招聘置顶管理 | 手动置顶/取消置顶热门招聘，支持设置置顶优先级和有效期 |
| 招聘黑名单 | 对恶意发布虚假招聘的用户加黑，禁止发布 |
| 公司库管理 | 高频发布招聘的企业信息库，支持企业认证信息管理和信用评级 |

**对应前端用户端模块**：「1. 本地招聘」

**后端接口需求**：
- `GET /api/admin/recruitments`（分页列表，带筛选参数）
- `GET /api/admin/recruitments/:id`
- `PUT /api/admin/recruitments/:id/status`（审核：approved / rejected / offline）
- `PUT /api/admin/recruitments/:id/top`（置顶设置，含有效期）
- `POST /api/admin/recruitment-blacklist`（黑名单管理）

### 模块五：数据统计

以下按业务场景细分统计维度：

| 统计维度 | 核心指标 | 图表类型 | 说明 |

| **内容发布分析** | | | |
| 各分类发布量 | 招聘/房源/便民/二手各分类发帖量 | 柱状图 + 饼图 | 分析用户偏好 |

| **用户活跃分析** | | | |
| 用户行为漏斗 | 打开小程序 → 浏览 → 发布 → 联系 → 发布成功 | 漏斗图 | 识别流失节点 |
| 用户留存率 | 日/周/月留存率 | 折线图 | 评估用户粘性 |
| 人均发布量 | 活跃用户人均发布信息数 | 数字卡片 | 衡量内容生态 |
| 用户活跃时段 | 分时段活跃用户数 | 热力图/折线图 | 指导推送策略 |
| **运营效果分析** | | | |
| 置顶订单销售额 | 置顶服务收入趋势 | 折线图 + 柱状图 | 评估变现效果 |
| 广告点击率（CTR） | 各广告位点击量/曝光量 | 柱状图 + 百分比 | 优化广告位配置 |
| 广告收入 | 按广告位/按时间统计 | 折线图 + 饼图 | 收入分析 |

| 客服触达率 | 用户咨询/投诉响应率 | 数字卡片 | 服务质量监控 |
| **告警与监控** | | | |
| 待审核信息数量 | 各分类待审条目数 | 告警卡片 | 红色标识高量分类 |
| 举报未处理数量 | 待处理举报数 | 告警卡片 | 超时需提醒 |
| 系统异常监控 | API错误率/响应时间 | 折线图 | 技术运维 |

**数据统计的实现路径**：
- **初期**：直接查询业务数据库进行聚合统计（招聘表count + 用户表count + 订单表sum等），简单SQL可实现80%的需求。
- **中期**：数据量大时，引入定时任务（如NestJS Cron）预聚合每日统计指标存入汇总表。
- **后期**：如需复杂多维度分析，再引入ClickHouse等OLAP数据库，或在运营中心配置自定义指标。

**数据表设计建议**：建议创建`daily_stats`汇总表，字段包括日期、分类、指标名、数值，每日定时任务更新，避免实时全表扫描带来的性能问题。

**后端接口需求**：
- `GET /api/admin/stats/dashboard`（核心指标汇总，支持时间范围）
- `GET /api/admin/stats/publication`（内容发布统计）
- `GET /api/admin/stats/user-activity`（用户活跃度统计）
- `GET /api/admin/stats/operation`（运营效果统计）
- `GET /api/admin/stats/revenue`（收入统计）


## 三、权限体系设计（RBAC）

Fantastic-admin提供了一套基于角色和路由的完整权限解决方案，可实现从页面到按钮的精细化管理。后台管理系统通常包含以下角色划分：

| 角色 | 权限范围 |
|------|----------|
| 超级管理员 | 全部权限，包括系统设置 |
| 内容审核员 | 可查看/审核所有分类信息（招聘/房源/便民/二手），无权限修改系统设置 |
| 运营人员 | 可置顶热门内容、管理广告位、发布资讯，无用户管理权限 |
| 客服人员 | 可查看用户列表、处理用户举报和投诉，无权审核内容 |

**Fantastic-admin权限实现机制**：框架内置了鉴权组件`<Auth>`和鉴权指令，可实现页面内特定模块的权限控制（如某个按钮只有审核员才能看到），支持在路由层面进行权限验证，通过配置文件按需启用权限模块。前端路由鉴权机制应确保未授权用户无法访问受限页面。

**后端权限实现**：NestJS通过全局或控制器级别的`RolesGuard`处理请求级权限校验，接口层面使用`@Roles()`装饰器标识所需的角色。整体设计包含User、Role、Permission三张实体表及user_role_relation、role_permission_relation两张关联表完成RBAC数据建模。此权限体系可无缝对接Fantastic-admin前端的路由及按钮鉴权，实现前后端权限统一。


## 四、API设计规范

后台管理模块的API命名遵循以下规范：

| 类型 | 路径格式 | 说明 | 权限要求 |
|------|----------|------|----------|
| 列表查询 | `GET /api/admin/{resources}` | 分页列表，支持筛选、排序 | 管理员/审核员 |
| 详情查询 | `GET /api/admin/{resources}/:id` | 单条资源详情 | 管理员/审核员 |
| 审核操作 | `PUT /api/admin/{resources}/:id/status` | 更新审核状态 | 审核员及以上 |
| 创建资源 | `POST /api/admin/{resources}` | 创建新资源（如资讯、广告） | 运营人员及以上 |
| 更新资源 | `PUT /api/admin/{resources}/:id` | 修改资源 | 运营人员及以上 |
| 删除资源 | `DELETE /api/admin/{resources}/:id` | 删除资源 | 管理员及以上 |
| 置顶操作 | `PUT /api/admin/{resources}/:id/top` | 设置置顶 | 运营人员及以上 |
| 批量操作 | `POST /api/admin/{resources}/batch/status` | 批量审核/下架 | 审核员及以上 |
| 统计查询 | `GET /api/admin/stats/{type}` | 获取统计数据 | 管理员/运营人员 |

**注意事项**：
- 使用`/admin`作为后台API前缀，与前端用户端API区分
- 所有后台API需要通过JWT认证 + RBAC权限双重校验
- 敏感操作（用户封禁、批量删除等）需记录操作日志
- 支持分页参数：`page`、`pageSize`、`sort`、`order`


## 五、开发优先级建议

按MVP（最小可行产品）开发顺序：
1. **第一轮（核心闭环）**：
   - 用户管理 + RBAC角色权限（基础）
   - 招聘管理（信息审核流程完整闭环）
   - 仪表盘（基础数据）
   - 登录/认证 + 操作日志
2. **第二轮（业务扩展）**：
   - 房源管理
   - 便民信息管理
   - 二手闲置管理
   - 分类管理（各业务分类动态配置）
3. **第三轮（运营变现）**：
   - 商家服务管理
   - 广告管理
   - 置顶订单管理
   - 资讯管理
   - 系统设置（公告、敏感词等）
4. **第四轮（数据驱动）**：
   - 完整的数据统计看板
   - 趋势分析和报表导出
   - 用户行为分析（漏斗、留存）

这样规划可以保证在最短时间内跑通核心的“信息发布→审核→展示”链路，后续业务模块和增值功能陆续追加。
